Verilog MIPS CPU
    • Collaborators

      Yuna Valade, Maximus Wickham, Dominic Clough, Aleksandar Limonov and Leonardo Garofalo

    • Duration

      November 25th - December 21st 2020

    • Languages and software involved

      Verilog, C++, Shell, Python

    • Overview

      This 6-person project was given as part of the Instruction Set Architecture and Compilers module in the 2nd year of Imperial's Electronic and Information Engineering course. The task was to create a working synthesizable 32-bit instructions MIPS multicycle processor using Verilog capable of executing 48 given instructions.
      Additionally, we were to write a testbench using a shell script file that tested the CPU for all 48 instructions.

    • My Roles

      I was tasked with creating the testbench for the CPU and writing most test cases. I found it really interesting researching how these shell scripts work and applying it in our own project taught me a lot and gave me confidence in using this in future tasks/projects.
      The testbench worked by:

      • Iterating through a folder of testcases and running them through an assembler which output the corresponding MIPS assembly language for those testcases in a separate folder,
      • Compiling the CPU files together into one and initialising the RAM with the input assembly language files,
      • Executing the compiled file of each test case and storing its output in a separate file,
      • Extracting each line from the output file with the prefix "RESULT : " into another file then removing that prefix and storing purely the result in a final output file,
      • Comparing the result of each test case with the pre-written reference result.

      One of the difficulties I faced with the testbench was writing about 5 testcases for each of the 48 instructions. I decided to write a script and C++ file that automatically generates values and set up testcases for me for most of the instructions which wouldn't need manual testcases written for them such as ADD instructions where the format for the instructions in the testcases is always the same. For instructions that did need checking for corner cases, they had extra test cases that were manually written.
      This allowed me to apply the shell script skills I learnt from writing the testbench. A single C++ file would take in the instruction we wanted testcases for and output the test instructions and the expected result all in a single file but with prefixes determining what they are. The shellscript would then extract the specific lines, remove the prefixes then place the instructions in a single file in one directory and the reference results in a single file in another directory.
